# Case Study Writer

## When to Use This Skill

Use this skill when the user wants to:
- Write a portfolio-ready case study
- Turn rough project notes into a structured narrative
- Tailor one project story for UX, PM, Tech/Data, or General audiences

## Pre-Generation Checklist

Confirm before writing:
- Portfolio type is identified: `UX`, `PM`, `Tech/Data`, or `General`
- Role and individual contribution are clear
- Timeline and team composition are known (or marked unknown)
- Problem statement and goals are defined
- Outcome metrics are available (or explicitly unavailable)

## Output Requirements

- Target length: `300-600` words
- Keep sections concise and concrete
- Prefer active voice and measurable outcomes
- If data is missing, state assumptions briefly instead of inventing facts

Structure output with these sections:
1. Project Overview
2. Problem
3. Goals
4. Approach
5. Execution
6. Outcome
7. Key Decisions
8. Learnings

## Recommended Section Length

| Section | Length |
| --- | --- |
| Overview | 3-4 lines |
| Problem | 4-6 lines |
| Goals | 2-4 lines |
| Approach | 5-8 lines |
| Execution | 5-10 lines |
| Outcome | 3-5 lines |
| Learnings | 2-3 lines |

## Portfolio Emphasis

Adjust focus based on audience:

- `UX`: research insights, usability issues, design rationale, iteration, validation
- `PM`: problem framing, prioritization trade-offs, roadmap influence, business impact
- `Tech/Data`: architecture constraints, technical decisions, system performance, scalability
- `General`: balanced narrative across problem, collaboration, execution, and outcomes

## Quality Bar

Before finalizing, verify:
- Contribution is explicit (`I led`, `I designed`, `I implemented`)
- Problem context is specific and meaningful
- Outcome includes at least one metric, or explains why metrics are unavailable
- Decision-making and trade-offs are visible
- Wording is concise and free of vague claims

## Common Failure Modes

- Vague impact claims without numbers
- Unclear ownership (team did X, but your role is missing)
- Missing problem context
- Overly long sections that hide the core story
